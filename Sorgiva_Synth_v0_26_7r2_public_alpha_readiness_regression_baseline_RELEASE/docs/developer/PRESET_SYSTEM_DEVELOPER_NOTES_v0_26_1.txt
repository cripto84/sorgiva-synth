Preset format
Il formato preset corrente è governato da PRESET_FORMAT_VERSION in preset-constants.js. I preset contengono metadata e parameters. Non tutti i controlli UI devono entrare in parameters.

Runtime boundary
I seguenti concetti non sono parte del timbro e non devono essere caricati come preset sonoro:
- scope-enabled;
- spectrum-enabled;
- MIDI channel filter runtime;
- MIDI Learn runtime;
- prefissi midi-* non sonori;
- category/originalCategory quando compaiono come vecchi metadati dentro parameters;
- filtri browser, ricerca, preferiti UI, selettori file, preview JSON;
- stato tab/pannelli;
- monitor diagnostici.

Regola pratica
Un preset deve poter cambiare il suono o un comportamento musicale esplicito. Non deve cambiare la comodità di lavoro dell'utente.

Factory preset
factory-presets.js contiene la libreria factory attiva. In v0.26.1 il count corrente è 135 e il file non viene modificato. La vecchia factory da 290 preset non deve rientrare.

User presets
Il preset manager può salvare copie utente, esportare JSON e importare preset/bank. Le chiavi legacy devono essere normalizzate, non applicate ciecamente.

A/B Compare e Morph
A/B Compare e Morph dipendono da snapshot coerenti dei parametri musicali. Non includere runtime/UI nelle snapshot sonore evita sorprese quando l'utente confronta o trasforma patch.

Randomizer
Il Randomizer deve lavorare su superfici sonore controllate, non su preferenze runtime. Non deve spegnere visualizer, cambiare MIDI Learn o alterare browser state.

Aggiungere un nuovo parametro preset-safe
Checklist:
1. È parte del suono o della performance musicale?
2. Ha un controllo HTML stabile?
3. Ha un default coerente?
4. È gestito da SynthXAudio o dal modulo runtime corretto?
5. Non rompe factory loading?
6. Non crea side-effect su MIDI/visualizer/browser?
7. È documentato?

Aggiungere un parametro da escludere
Se un controllo non deve entrare nel preset:
- aggiungerlo a EXCLUDED_IDS o RUNTIME_PRESET_IDS se appropriato;
- verificare import/export;
- testare caricamento factory e user preset;
- documentare il motivo nel QA report.

Regola per factory count
Ogni QA deve contare:
- numero factory preset;
- ID unici;
- nomi unici;
- categorie visibili/tassonomia;
- categorie orfane = 0.
