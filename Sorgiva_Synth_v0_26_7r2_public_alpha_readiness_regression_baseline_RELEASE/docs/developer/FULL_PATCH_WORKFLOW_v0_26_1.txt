Sorgiva Synth — FULL / PATCH Workflow v0.26.5

Regola utente
Ogni avanzamento deve fornire almeno un FULL bundle aggiornato. La PATCH è aggiuntiva e serve a chi vuole applicare solo le differenze, ma non deve essere l'unico output.

FULL bundle
Il FULL deve contenere l'intero progetto in una cartella radice con nome versione. Deve essere apribile/consultabile senza ricostruire parti da vecchie build.

PATCH bundle
La PATCH deve contenere solo i file modificati o aggiunti rispetto alla baseline immediatamente precedente. I percorsi devono essere relativi alla root del progetto.

Test patch->full
Procedura:
1. Estrarre la baseline precedente in una cartella temporanea.
2. Applicare la PATCH sovrascrivendo i file corrispondenti.
3. Confrontare il risultato con il FULL v corrente.
4. Ignorare solo metadati esterni allo zip se non fanno parte del progetto.
5. Se il diff non è identico, la PATCH non è valida.

Cosa includere sempre nei cambi versione
- README.txt.
- CHANGELOG.txt.
- VERSION.
- VERSION.txt.
- index.html se mostra la versione o testo di stato.
- src/state/state.js se appVersion cambia.
- src/main.js se log o messaggi versione cambiano.
- docs/release-notes/QA_*.txt.
- metriche JSON se utili.

Cosa evitare
- Non mettere report lunghi nella root.
- Non includere backup temporanei.
- Non includere cartelle __MACOSX, .DS_Store, node_modules, cache o file di lavoro.
- Non modificare factory library se la fase è solo docs/metadata.
- Non toccare audio-context.js salvo bug reale.

Nomi consigliati
FULL:
Sorgiva_Synth_vX_Y_Z_nome_fase_FULL.zip

PATCH:
Sorgiva_Synth_vX_Y_Z_nome_fase_PATCH.zip

Nota
Il confronto patch->full è particolarmente importante perché l'utente usa spesso le build come baseline di lavoro nelle istanze successive.


Nota rename v0.26.4d
Il vecchio prefisso SynthX_Rebuild resta valido solo per archivi storici pre-rename. Le nuove build pubbliche devono usare Sorgiva_Synth_* per evitare ambiguita di branding.
